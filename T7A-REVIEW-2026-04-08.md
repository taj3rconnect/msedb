## Product Health Report — MSEDB

### Executive Summary

MSEDB has a functioning core — it connects to M365 mailboxes, syncs email events, detects patterns, and stages actions for approval. However, the application has critical security vulnerabilities (zero-auth MongoDB and Redis exposed on the network, Docker socket mounted in the backend container), virtually no test coverage (2 test files across 38,000+ lines), and a rate limiter that never fires because it is mounted after all routes. The product has drifted from its PRD into a general-purpose inbox client, diluting engineering effort away from the core pattern-to-rule pipeline that is the product's reason to exist. Before any new features ship, the security holes must be closed, the safety-critical code paths (action execution, rule matching) must be tested, and the codebase must be stabilized.

### P0 — Ship Blockers

- [ ] **MongoDB zero authentication** — `docker-compose.yml` (mongo service) — **S** — flagged by Security, DB Architect, System Architect. MongoDB is bound to all interfaces on port 27020 with no `--auth`, no user/password. MSAL token caches and all email event data are readable by anyone who can reach the port.

- [ ] **Redis zero authentication** — `docker-compose.yml` (redis service) — **S** — flagged by Security, DB Architect. Port 6382 exposed, no `requirepass`. An attacker can inject BullMQ jobs or flush the cache. Also change eviction policy from `noeviction` to `allkeys-lru` — when Redis fills its 384MB cap, ALL writes (including BullMQ job enqueues) hard-fail.

- [ ] **Rate limiter mounted AFTER routes — zero protection** — `server.ts:127-128` — **S** — flagged by Security, System Architect. The rate limiter middleware is `app.use()`'d after all route registrations, so it never intercepts any request. Move it above route mounting.

- [ ] **Docker socket mounted in backend container** — `docker-compose.yml:22` — **S** — flagged by Security, System Architect. `/var/run/docker.sock` is volume-mounted and appuser is in the docker group. Container escape vector. Remove the mount entirely.

- [ ] **JWT/encryption secrets default to empty string** — `config/index.ts:46-48` — **S** — flagged by Security. If `JWT_SECRET` or `ENCRYPTION_KEY` env vars are unset, the app starts silently with forgeable tokens. Add startup validation that crashes the process.

- [ ] **Zero tests on safety-critical paths** — `actionExecutor.ts`, `ruleEngine.ts`, `stagingManager.ts`, `stagingProcessor.ts` — **M** — flagged by QA. These files execute destructive operations (delete/move emails) and determine which emails get acted on. A single bug here silently deletes a user's email.

### Cross-Cutting Concerns

**1. No defense in depth.** MongoDB, Redis, JWT secrets, and rate limiting all have the same class of failure: "works fine in dev, catastrophic in production." A single checklist of "things that must have auth/secrets" would have caught all four.

**2. God Files are the root cause of multiple symptoms.** `mailbox.ts` (1,805 lines), `InboxPage.tsx` (2,659 lines), and `patternEngine.ts` contribute to untestability, performance issues (N+1 queries hidden in deep nesting), and inability to onboard new contributors.

**3. Test debt is compounding.** At 2 test files for 38,000+ lines, every new feature increases the risk surface. The most dangerous code (action execution, rule matching) has zero coverage.

**4. Product scope drift.** The PRD describes a pattern detection and rule creation tool. The codebase includes a full inbox client, email composer, calendar sync, and contact management.

### What's Working Well

1. **Encrypted token storage (AES-GCM)** — MSAL tokens encrypted at rest
2. **TTL indexes on core models** — Prevents unbounded growth on main collections
3. **Compound dedup index on EmailEvent** — Prevents duplicate event ingestion
4. **Evidence array validator on patterns** — Captures reasoning for user trust
5. **Staging/approval model** — No action taken without user approval; safety-first design
6. **Docker Compose setup works** — All 4 containers come up healthy with resource limits and health checks

### Hard Truths

1. **This application is not shippable to anyone but the developer.** Zero-auth MongoDB on a network-accessible port with MSAL token caches inside it is a data breach waiting to happen.

2. **You are building features on top of code you cannot prove works.** The action executor deletes real emails. The rule engine decides which emails to act on. Neither has a single test.

3. **The product has lost focus.** A full inbox client with compose, calendar sync, and contact management is not what the PRD describes. The differentiation is pattern-to-rule automation, not being another email client.

4. **The God Files are a velocity wall.** You cannot meaningfully test, review, or parallelize work on a 2,659-line React component or an 1,805-line route file.

5. **The "it works on my machine" configuration will not survive deployment.** Hardcoded IPs, hardcoded emails, empty-string secret defaults, no CI pipeline — every one is a deployment failure waiting to happen.

---

### The 5 Questions You Must Answer Before Shipping

1. **Who is the first real user, and what is the minimum they need?** Pick the one workflow that delivers value — pattern detection to rule creation — and ship that completely before anything else.

2. **What happens when the action executor has a bug?** Define blast radius controls: max emails per action run, undo window, alerting on unusual deletion volumes. Then test them.

3. **Can you recover from a MongoDB failure today?** No backups, no replica set, no PITR. If the volume corrupts, all data is gone.

4. **Are you building for one user or many?** Hardcoded emails and single-tenant assumptions suggest a personal tool. If multi-tenant is the goal, the architecture needs to change now.

5. **What is your testing bar for code that touches real user data?** The current answer is "none." Define the minimum and commit to it before writing another feature.
